### 对话 7: 前端补齐 app.js / api.js / utils.js
**时间**: 2026-06-11 下午
**Commit**: `4798af0 feat(frontend): 补齐前端交互、API 通信与工具函数`

**用户输入**:
> app.js / api.js / utils.js 写一下，让前端能跑通。

**AI 理解**:
- app.js: 状态管理、事件处理、Markdown 拆分、结果渲染
- api.js: HTTP 通信、错误处理、超时控制
- utils.js: DOM 工具、Markdown 渲染、剪贴板、防抖

**关键决策**:
- **状态集中**: `STATE` 对象统一管理 (避免散落的全局变量)
- **DOM 缓存**: 启动时一次 `cacheDom()` , 避免反复查询
- **Markdown 拆分用字符串切分** (而非 parseTree, 服务端格式固定, 简单方案已够)
- **双重 AbortController**: 超时 + 用户主动取消都能触发
- **错误码体系**: 前后端共用 `INVALID_INPUT` / `TIMEOUT` / `NETWORK_ERROR` 等
- **占位符处理**: 输入为空时显示"未提供" / "未指定" 而非留空

**踩坑**:
- 当时用 `<script type="module">` 试过, 但需要 HTTP server 跑, 与"零构建"目标冲突
- **回退方案**: 挂全局对象 + 顺序加载

### 对话 13: 修复 script 加载顺序
**时间**: 2026-06-11 晚上
**Commit**: `ef166d8 fix(frontend): 补齐 script 加载顺序 + 优化 JS 引用方式`

**用户输入**:
> 我用浏览器打开页面, 控制台报错 "PitchForgeAPI is not defined", 排查一下。

**AI 理解**:
- `app.js` 解构了 `PitchForgeAPI`, 但加载顺序错了
- 同时, `app.js` 和 `api.js` 都用了 IIFE 包装, 直接 `<script src>` 引用没问题
- 修复: 确保 utils.js → api.js → app.js 顺序

**关键决策**:
- **明确加载顺序** 写在 `index.html` 注释里, 避免后续误调
- **解构方式** 改回直接 `const { ... } = PitchForgeUtils;`, 避免 IIFE 解析问题
- **保留全局对象** 模式 (而非 ES Module), 与"零构建"目标一致

**踩坑教训**:
- 全局对象挂载 + 顺序加载, 比 `<script type="module">` 简单, 但**顺序错了就是 undefined**
- 必须在 HTML 注释里写明顺序, 团队成员才不会乱动

## 💡 关键洞察记录

### 产品定位洞察
- PitchForge 不是替用户编造项目, 而是帮助用户整理真实想法
- 核心价值：降低项目表达门槛, 提高现场交付效率
- 目标场景：60-90 分钟的黑客松/Workshop 实操后
- 战略锚点: 不做独立产品, 做"阿里云百炼的最佳实践案例"

### 技术选型洞察
- 推荐：百炼工作流 + 简单网页 (而非单 Agent)
- 实际落地: **单次 Application Call + 1 聚合 Prompt** (MVP 取舍)
- 原因：调试链路短、上线快、未来可平滑迁移到工作流
- MVP 阶段保持轻量, 避免过度设计

### 用户体验洞察
- 用户痛点：表达结构混乱、路演稿难写、Demo 展示无流程、害怕评委追问
- 解决方案：结构化生成 4 个核心输出 (项目结构、路演稿、Demo 流程、评委问答)
- 设计原则：简洁、专业、高效、Builder 工具感

### 路演执行洞察
- **80% 的路演失败 = 网络/设备问题**, 不是产品问题
- 备用输出文件 (`sample-output.md`) 是"专业感"的一部分
- 评委追问要在脚本里准备 6 个, 实战往往只问 1-2 个
- **1 分钟话术要练 5 遍以上**, 控制时长在 60-90 秒

### 协作/工程洞察
- 全局对象 + 顺序加载, 比 ES Module 简单, 但**顺序错了 = undefined**
- 错误码体系要前后端共用, 否则翻译成本高
- Mock 兜底比"等到 API 调通再跑"重要, 让 Demo 不依赖外部
- 注释要写"为什么" 而非"做什么" (代码自解释做什么)
