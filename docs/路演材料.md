## 🎤 1 分钟产品介绍话术

> 目标时长：60-90 秒。语气自然、有节奏感。**别背稿**。

```
大家好，我做的是 PitchForge，一个基于阿里云的 AI 路演工坊。

我观察到，在黑客松和 AI Workshop 中，很多 Builder 并不是没有想法，
也不是完全做不出 Demo，而是很难在短时间内把项目讲清楚。

尤其是在 60 到 90 分钟的实操之后，大家还要马上准备项目结构、
路演稿、Demo 展示流程和评委问答，这个过程非常紧张。

所以我做了 PitchForge。用户只需要输入一个粗略的项目想法，
系统就可以自动生成项目结构、1 分钟路演稿、Demo 展示流程
和评委可能追问。

这个工具的价值不是替用户编造项目，而是帮助用户把真实想法
快速打磨成清晰、可信、可展示的表达材料。

在技术实现上，我使用阿里云搭建 AI 应用，将项目表达过程
拆成结构化分析、路演稿生成、Demo 流程生成和评委问答模拟几个步骤。

未来，PitchForge 可以继续扩展成 Hackathon Copilot，支持 
PRD 生成、README 生成、项目提交材料检查和赛后复盘，
帮助更多 Builder 从 idea 更高效地走向可展示作品。
```

### 练习要点
- [ ] 控制时间在 60-90 秒（用手机计时器反复练）
- [ ] 语速自然、抑扬顿挫（"我观察到" 用降调，"PitchForge" 重音）
- [ ] 突出核心价值（"不是替用户编造项目" 这句要清楚）
- [ ] 提及技术亮点（"阿里云"、"AI 工作流" 这两个词必须出现）

## 🎬 Demo 展示流程（2-3 分钟）

### Step 1：展示首页（15 秒）

- [ ] 打开浏览器，输入 `http://localhost:3000`
- [ ] 介绍布局："左边输入，右边输出，4 个卡片会依次亮起"
- 话术：`"这是 PitchForge 的主页。左边是输入区，右边是输出区。核心操作只有一个：填一个想法，点一下生成。"`

### Step 2：输入示例（30 秒）

- [ ] 点击底部"填充示例输入"按钮
- [ ] 简要说明 5 个字段
- 话术：`"我已经准备好了 PitchForge 自己的项目想法。我们点一下生成，看 AI 工作流会怎么跑。"`

### Step 3：点击生成（30 秒）

- [ ] 点击"生成我的路演包"按钮
- [ ] 展示进度条 4 个阶段
- 话术：`"系统在阿里云上调用 AI 工作流，分 4 步生成：先整理项目信息，再写路演稿，再设计 Demo 流程，最后准备评委问答。这个过程大概需要 1 分钟。"`

### Step 4：展示输出（60 秒）

- [ ] 依次滚动 4 个卡片
- [ ] **重点强调** 项目结构和 1 分钟路演稿
- 话术：`"看，这就是 4 个模块：项目结构清晰、1 分钟路演稿可以直接念、Demo 流程 3-5 步、评委问答 5 个高频问题。每块都有复制按钮。"`

### Step 5：总结价值（20 秒）

- [ ] 邀请评委现场体验
- 话术：`"这就是 PitchForge。如果评委有兴趣，可以现场扫码体验。链接我会贴到群里。"`

### 备用方案 ⚠️

如果现场网络异常：
1. 打开 `docs/sample-output.md`（已预生成）
2. 展示预先生成的 4 个模块
3. 强调"这是离线版本，演示逻辑完全一致"

## ❓ 评委问答准备

### Q1：和普通 Chatbot 有什么区别？（差异化类）

**回答思路**：
- 场景化 vs 自由对话
- 输出结构化 vs 输出不稳定
- 可直接用 vs 需二次加工

**参考回答（30 秒）**：
```
普通 Chatbot 是自由对话，输出不稳定，每次长度、格式、风格都不同。
PitchForge 面向路演准备这个具体场景，把生成过程拆成 4 个独立模块。
输出格式固定，可直接复制到 PPT 或答辩稿里。
简单说，Chatbot 是聊天工具，PitchForge 是路演工具。
```

---

### Q2：为什么一定要用 AI 来做？（价值类）

**回答思路**：
- 效率提升：2-3 小时 → 1-2 分钟
- 标准化输出
- 适合时间紧迫场景

**参考回答（20 秒）**：
```
人工整理项目结构和路演稿通常需要 2-3 小时，
而 AI 只需 1-2 分钟，且输出结构标准化。
这对时间紧张的黑客松场景非常重要。
```

```
核心用户是黑客松参与者，他们只有 60-90 分钟实操时间，
路演准备时间通常不足 20 分钟。
延伸用户包括独立开发者、AI Workshop 学员、早期项目 Builder，
他们都需要在短时间内完成项目表达材料。
```

---

### Q6（备用）：技术实现细节？（技术深度）

**回答思路**：
- Application Call API
- 单次调用 + 聚合 Prompt
- 后续可升级到工作流模式

**参考回答（40 秒）**：
```
后端调用阿里云的 Application Call API。
前端是 5 个字段，后端做二次校验后构造请求体，
传给的 app ID，用 system prompt 引导模型按 4 个模块输出。
Prompt 用了 few-shot + 严格 Markdown 格式约束。
当前是单次调用，输出稳定性还可以更好，
下一步会迁移到工作流，把 4 个模块拆成独立节点，
输出会更稳定，也更便于调试。
```
