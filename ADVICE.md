## 🎯 技术选型建议 (设计阶段)

### Q1: 前端应该用原生 HTML/CSS/JS 还是框架 (React/Vue)?

**建议**: MVP 阶段使用 **原生 HTML/CSS/JS**

**理由**:
1. ✅ **开发速度快**: 无需配置构建工具, 直接编写即可
2. ✅ **学习成本低**: 90 分钟内可完成, 无需处理框架复杂度
3. ✅ **部署简单**: 静态文件可直接托管 (Vercel/Netlify)
4. ✅ **易于演示**: 代码结构简单, 现场演示更易理解

**何时考虑框架**:
- 需要组件复用 (3+ 相似页面)
- 需要状态管理 (复杂用户交互)
- 团队协作开发 (多人同时修改)
- 长期维护迭代 (P2/P3 阶段)

---

### Q2: 后端是否需要 Node.js/Express？能否直接前端调用百炼 API?

**建议**: MVP 阶段推荐 **Node.js/Express 后端**

**理由**:
1. 🔒 **安全性**: API Key 不暴露在前端
2. 🛡️ **控制力**: 可添加限流、日志、错误处理
3. 📦 **扩展性**: 后续添加数据库、用户认证更方便
4. 🎓 **演示价值**: 展示完整前后端架构

**直接前端调用的风险**:
- ❌ API Key 暴露 (安全风险)
- ❌ CORS 问题 (需配置跨域)
- ❌ 无法限流 (可能被恶意调用)
- ❌ 错误处理弱 (用户体验差)

---

### Q3: 百炼工作流 vs 单 Agent，如何选择？

**建议 (设计阶段)**: 选择 **百炼工作流**
**实际落地**: 采用 **单次 Application Call + 1 个聚合 Prompt**

**对比分析**:

| 维度 | 工作流 | 单次 Application Call |
|------|--------|------------------------|
| 输出稳定性 | ✅ 高 (多节点独立控制) | ⚠️ 中 (依赖聚合 Prompt) |
| 现场展示 | ✅ 清晰 (展示多步骤) | ⚠️ 单一 (一次调用) |
| 调试难度 | ⚠️ 中 (需配置多节点) | ✅ 低 (单次 Prompt 调优) |
| 扩展性 | ✅ 高 (可添加节点) | ⚠️ 中 (Prompt 越来越长) |
| 开发时间 | ⚠️ 20-30 分钟 | ✅ 10-15 分钟 |
| 配置成本 | ⚠️ 高 (需在百炼控制台配置) | ✅ 低 (只需 App ID + API Key) |

**结论**:
- 设计阶段推荐工作流 (因为理论上更稳定)
- 实施阶段选择单次调用 (因为 MVP 上线更快)
- 后续可平滑迁移: 4 个独立 Prompt (`prompts/*.md`) 已经准备好, 直接搬到百炼工作流即可

**为什么这样取舍**:
- 评委问"为什么不用工作流"时, 可以说"我们做了 P1 升级计划, MVP 阶段先验证产品价值"
- **避免过度工程**: 90 分钟内, 调通 5 个节点比调通 1 个节点难 5 倍

## 🎨 设计规范建议

### Q4: 页面布局如何设计？

**建议**: 采用 **左右分栏布局**

**布局结构**:
```
┌─────────────────────────────────────┐
│         顶部：产品标题 + 副标题       │
├──────────────┬──────────────────────┤
│   左侧输入区   │      右侧输出区        │
│              │                      │
│  - 项目想法    │  - 项目结构           │
│  - 目标用户    │  - 1 分钟路演稿       │
│  - 当前进度    │  - Demo 展示流程      │
│  - 路演时长    │  - 评委问答           │
│  - 想突出的亮点 │                      │
│              │                      │
│  [生成按钮]    │  [复制按钮] [重新生成]  │
├──────────────┴──────────────────────┤
│         底部：示例输入 + 使用说明       │
└─────────────────────────────────────┘
```

**设计理由**:
- ✅ 清晰的信息架构 (输入 → 输出)
- ✅ 符合用户心智模型
- ✅ 便于现场演示 (从左到右)
- ✅ 响应式友好 (移动端可上下堆叠)

### Q5: 色彩方案如何应用？

**建议**: 基于 PRD 色彩规范, 采用 **功能导向配色**

**色彩应用指南**:

| 元素 | 颜色 | 色值 | 用途 |
|------|------|------|------|
| 背景 | 浅灰 | #F7F8FA | 页面整体背景 |
| 主色 | 靛蓝 | #4F46E5 | 按钮、链接、强调 |
| 强调色 | 翠绿 | #10B981 | 成功状态、高亮 |

| 卡片背景 | 白 | #FFFFFF | 内容区块 |

**使用原则**:
1. 主色用于 **交互元素** (按钮、链接)
2. 强调色用于 **成功/完成状态**
3. 文本色用于 **内容阅读**
4. 辅助文本用于 **次要信息**
5. 避免超过 3 种主色调

## 🔧 开发实践建议

### Q6: 如何处理 AI 生成的 Loading 状态？

**建议**: 采用 **渐进式 Loading + 文字提示**

**实施代码** (`frontend/js/app.js`):
```javascript
const steps = [
  { p: 15, t: '正在整理项目信息...', d: 0 },
  { p: 35, t: '正在生成路演稿...', d: 3000 },
  { p: 60, t: '正在设计 Demo 流程...', d: 6000 },
  { p: 85, t: '正在准备评委问答...', d: 9000 },
  { p: 95, t: '正在整理最终结果...', d: 12000 },
];
```

**为什么用模拟进度**:
- 工作流是黑盒调用, 前端拿不到真实进度
- 用户需要看到"系统在干活", 否则会以为卡死
- 模拟 5 阶段文案与实际输出模块对齐, 体感更真实

**为什么留 600ms 延迟**:
- 让用户看到"100%"的完成状态
- 太快会"嗖一下"就完了, 显得没干活
- 太慢会让人觉得慢

### Q7: 如何处理 API 调用失败？

**建议**: 采用 **三层防护 + 业务错误码体系**

**实施代码** (`backend/services/bailianService.js`):
```javascript
function normalizeError(err) {
  if (err.code === 'ECONNABORTED') return createBailianError('TIMEOUT', '...');
  if (!err.response) return createBailianError('NETWORK_ERROR', '...');
  if (status === 401) return createBailianError('UNAUTHORIZED', '...');
  // ... 完整错误码体系
}
```

**业务错误码清单**:
| 错误码 | HTTP | 含义 |
|--------|------|------|
| INVALID_INPUT | 400 | 输入参数无效 |
| UNAUTHORIZED | 401 | API Key 错误 |
| RATE_LIMITED | 429 | 请求过于频繁 |
| TIMEOUT | 504 | API 超时 |
| NETWORK_ERROR | 502 | 网络异常 |
| SERVICE_UNAVAILABLE | 503 | 服务繁忙 |
| INTERNAL_ERROR | 500 | 兜底错误 |

**错误提示示例**:
```
❌ 生成失败，请稍后重试。
可能原因：网络不稳定、服务繁忙
[重试按钮] [查看示例输出]
```

**关键设计**:
- 错误码 **前后端共用** (前端 utils.js 里的 `getErrorMessage` 直接映射)
- 永远不暴露 `err.stack` 给前端
- 调试模式下可以返回 `detail` 字段 (便于开发)

### Q8: 如何优化 Prompt 输出质量？

**建议**: 采用 **结构化 Prompt + Few-shot 示例**

**Prompt 三件套**:
1. **角色定位**: "你是一个资深的黑客松路演教练"
2. **输出模板**: 严格 Markdown 格式, 4 个二级标题
3. **Few-shot**: 1-2 个示例输入/输出

**约束语模板**:
```
输出要求：
- 使用中文
- 内容务实，不要夸大
- 不要编造未完成的功能
- 每个模块控制在 X 字内
```

**为什么不堆更多约束**:
- 约束越多, 模型"顾此失彼"概率越高
- 真正有效的约束 = **角色** + **格式** + **示例** + **3-5 条硬约束**

## 🏗️ 实施阶段建议 (MVP 当天沉淀)

### Q11: 单次调用 vs 5 节点工作流, 实施时怎么选?

**建议**: **单次 Application Call + 1 个聚合 Prompt**

**理由**:
- 90 分钟内调通 5 个节点, 平均每个 18 分钟, 出问题根本来不及排查
- 聚合 Prompt 用 Few-shot 也能拿到 70-80% 的稳定性
- 调试链路短: 一个 `curl` 就能验证后端

**预留迁移路径**:
- 4 个独立 Prompt (`prompts/{idea_structure,pitch_script,demo_flow,qa}.md`) 保留
- 后续迁移 = 把这 4 个 Prompt 搬到百炼工作流节点, 加 1 个聚合节点
- **零代码改动**, 只改 `bailianService.js` 的调用方式

**关键话术** (路演时):
> "MVP 阶段我们用了单次 Application Call, 因为我们的核心假设是'产品价值是否成立', 调通工作流是次要矛盾。P1 阶段我们会升级到 5 节点工作流, 让输出稳定性从 80% 提到 95%。"

### Q13: 前端要不要用 `<script type="module">`?

**建议**: **不要, 用全局对象 + 顺序加载**

**对比**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| ES Module (`type="module"`) | 标准、清晰 | 必须 HTTP server, 不能 `file://` 直接打开 |
| 全局对象 + 顺序加载 | 简单、零构建 | 顺序错了 = undefined, 需要注释约束 |

**为什么选全局对象**:
- "零构建" 是 MVP 硬指标
- 团队成员不会乱动 script 顺序 (有 HTML 注释约束)
- 调试时 Chrome DevTools 直接断点, 不需要 source map

**关键注释** (`index.html`):
```html
<!-- 工具函数库 (必须在 app.js 之前加载) -->
<script src="js/utils.js"></script>
<script src="js/api.js"></script>
<script src="js/app.js"></script>
```

**踩坑教训**:
- `app.js` 解构 `PitchForgeAPI` 时, 加载顺序错了 = `Cannot destructure ... of undefined`
- 当时用 5 分钟排查, **写到 HTML 注释里** 避免下次再踩

### Q14: 后端限流怎么配?

**建议**: `express-rate-limit`, 15 min / 30 次

**为什么是 30 次**:
- 真实用户 1 次提交算 1 次, 30 次 = 每天 30 个不同用户
- 黑客松现场 50 人/天, 不会触发限流
- 防滥用, 不防正常使用

**配置位置** (`backend/server.js`):
```javascript
app.use('/api/', apiLimiter); // 仅 API 路由
```

**注意**:
- 仅对 `/api/` 前缀生效, 静态文件不限流
- 错误响应统一格式 (`success: false` + `error.code: RATE_LIMITED`)
- 标准响应头 (`RateLimit-Remaining`) 让前端能预判

### Q15: Markdown 渲染要不要引入库?

**建议**: **不引入, 自己写最小渲染器**

**原因**:
- 服务端输出格式固定 (4 个二级标题 + 列表)
- 引入 `marked` (30KB) 或 `markdown-it` (90KB) 性价比低
- 自己写 50 行就能覆盖 80% 场景

**实施** (`frontend/js/utils.js`):
```javascript
function renderMarkdown(md) {
  return md
    .replace(/^### (.*$)/gim, '<h3>$1</h3>')
    .replace(/^## (.*$)/gim, '<h2>$1</h2>')
    .replace(/^- (.*$)/gim, '<li>$1</li>')
    .replace(/\*\*(.*?)\*\*/g, '<strong>$1</strong>');
}
```

**进阶方案 (P1)**:
- 用 `DOMPurify` 清洗 XSS (现在直接 `innerHTML` 有风险)
- 或用 `marked` + `DOMPurify` 组合 (多 60KB, 安全性高 10 倍)

### Q16: 错误码应该用 HTTP 状态码还是自定义业务码?

**建议**: **业务错误码 + HTTP 状态码 双层映射**

**为什么不只用 HTTP**:
- 503 既可能是"服务繁忙"也可能是"维护中", 业务码能区分
- 前端 `getErrorMessage` 直接 `switch(code)`, 翻译成本最低
- 监控/日志系统按业务码聚合更准确

**为什么不只用业务码**:
- HTTP 客户端 (`fetch` / `axios`) 默认按 status 抛错
- 负载均衡/网关按 status 路由 (5xx 重试, 4xx 不重试)
- 工具链友好

**实施映射表** (`backend/routes/generate.js`):
```javascript
function mapErrorCodeToStatus(code) {
  return {
    INVALID_INPUT: 400,
    UNAUTHORIZED: 401,
    RATE_LIMITED: 429,
    TIMEOUT: 504,
    NETWORK_ERROR: 502,
    SERVICE_UNAVAILABLE: 503,
    INTERNAL_ERROR: 500,
  }[code] || 500;
}
```

### Q19: 评委说"你这个跟 ChatGPT 有什么区别"怎么办?

**建议**: 强调"场景化" + "结构化" + "可路演"

**参考回答 (30 秒)**:
```
普通 ChatGPT 是聊天工具, 你问什么它答什么, 格式不固定。
PitchForge 是路演工具, 你只输入项目想法, 它会按 4 个固定模块输出:
项目结构、路演稿、Demo 流程、评委问答。
```

**强调 3 个差异**:
1. **场景化** (路演 vs 通用)
2. **结构化** (固定 4 模块 vs 自由回答)
3. **可路演** (直接复制用 vs 二次加工)

### Q20: 现场网络挂了, 备用方案怎么用得不尴尬?

**建议**: **提前铺垫 + 平滑切换**

**铺垫话术** (路演前 30 秒):

- 强调"逻辑一致" (降低评委疑虑)

## ⚠️ 反例与反思 (下次不要再犯)

### Q21: 不要在 MVP 阶段追求"完美架构"

**反例**:
- 设计了 5 节点工作流, 实际 90 分钟根本调不通
- 抽象了"AI Service 接口" + "多平台适配器", 但只有 1 个平台

**反思**:
- 架构是**演化出来的**, 不是**设计出来的**
- 90 分钟内能跑通, 比 90 分钟内"架构优雅"重要 10 倍
- 真正的可扩展性 = **预留接口** + **零迁移成本**, 不是"过度抽象"

---

### Q22: 不要让 AI 替你"补"用户没说的内容

**反例**:
- 用户只输入"做个 AI 路演助手", AI 编造"目标用户: 大学生, 进度: 已上线"
- 路演时评委发现"你这些数据哪来的?"

**正确做法**:
- 选填字段为空时, **保留为空**, 不要让 AI 编
- Prompt 约束: "用户未提供的内容, 显示'未提供' 或'未指定', 不要自行补充"
- 路演时主动说"我故意没填, 看 AI 会不会瞎编"

---

### Q23: 不要在路演材料里写"宏大愿景"

**反例**:
- "我们要做黑客松操作系统, 覆盖全生命周期, 三年内服务 10 万用户"
- 评委: "你这 90 分钟做的东西, 跟这愿景有关系吗?"

**正确做法**:
- 1 分钟话术里**只讲已经做完的**, 未来扩展 1 句话带过
- 让"做出来的东西" 说话, 而不是"PPT 上的愿景"
- 评委更相信"这个 MVP 能跑", 不相信"三年内改变世界"

---

### Q24: 不要把 `node_modules` 提交到 Git

**反例**:
- 第一次 commit 时, `node_modules` 200MB 全部进 git
- 后续每次 pull 慢 30 秒

**正确做法**:
- `.gitignore` 一定要有 `node_modules/`
- `package-lock.json` 提交, 团队可复现
- 协作时 `npm install` 安装依赖

---

### Q25: 不要在 `.env` 写明文 API Key 后直接 commit

**反例**:
- `.env` 里写 `API_KEY=sk-1234567890abcdef`, 提交到公开仓库
- 别人 5 分钟内刷光你的额度

**正确做法**:
- `.gitignore` 加 `.env`
- 提供 `.env.example` 作为模板
- 真实密钥只放在部署平台 (Vercel/阿里云 FC) 的环境变量里
