## 🎯 技术选型建议

### Q1: 前端应该用原生 HTML/CSS/JS 还是框架 (React/Vue)?

**建议**: MVP 阶段使用 **原生 HTML/CSS/JS**

**理由**:
1. ✅ **开发速度快**: 无需配置构建工具，直接编写即可
2. ✅ **学习成本低**: 90 分钟内可完成，无需处理框架复杂度
3. ✅ **部署简单**: 静态文件可直接托管 (Vercel/Netlify)
4. ✅ **易于演示**: 代码结构简单，现场演示更易理解

**何时考虑框架**:
- 需要组件复用 (3+ 相似页面)
- 需要状态管理 (复杂用户交互)
- 团队协作开发 (多人同时修改)
- 长期维护迭代 (P2/P3 阶段)

---

### Q2: 后端是否需要 Node.js/Express？能否直接前端调用百炼 API?

**建议**: MVP 阶段推荐 **Node.js/Express 后端**

**理由**:
1. 🔒 **安全性**: API Key 不暴露在前端
2. 🛡️ **控制力**: 可添加限流、日志、错误处理
3. 📦 **扩展性**: 后续添加数据库、用户认证更方便
4. 🎓 **演示价值**: 展示完整前后端架构

**直接前端调用的风险**:
- ❌ API Key 暴露 (安全风险)
- ❌ CORS 问题 (需配置跨域)
- ❌ 无法限流 (可能被恶意调用)
- ❌ 错误处理弱 (用户体验差)

---

### Q3: 百炼工作流 vs 单 Agent，如何选择？

**建议**: 选择 **百炼工作流**

**对比分析**:

| 维度 | 工作流 | 单 Agent |
|------|--------|----------|
| 输出稳定性 | ✅ 高 (多节点独立控制) | ⚠️ 中 (依赖单次生成) |
| 现场展示 | ✅ 清晰 (展示多步骤) | ⚠️ 单一 (黑盒调用) |
| 调试难度 | ⚠️ 中 (需配置多节点) | ✅ 低 (单次 Prompt) |
| 扩展性 | ✅ 高 (可添加节点) | ⚠️ 低 (Prompt 会越来越长) |
| 开发时间 | ⚠️ 20-30 分钟 | ✅ 10-15 分钟 |

**结论**: 工作流更适合 PitchForge，因为：
- 输出结构稳定 (4 个独立输出)
- 现场演示更有说服力 (展示 AI 工作流)
- 后续扩展更方便 (添加新节点)

## 🎨 设计规范建议

### Q4: 页面布局如何设计？

**建议**: 采用 **左右分栏布局**

**布局结构**:
```
┌─────────────────────────────────────┐
│         顶部：产品标题 + 副标题       │
├──────────────┬──────────────────────┤
│   左侧输入区   │      右侧输出区        │
│              │                      │
│  - 项目想法    │  - 项目结构           │
│  - 目标用户    │  - 1 分钟路演稿       │
│  - 当前进度    │  - Demo 展示流程      │
│  - 路演时长    │  - 评委问答           │
│  - 想突出的亮点 │                      │
│              │                      │
│  [生成按钮]    │  [复制按钮] [重新生成]  │
├──────────────┴──────────────────────┤
│         底部：示例输入 + 使用说明       │
└─────────────────────────────────────┘
```

**设计理由**:
- ✅ 清晰的信息架构 (输入 → 输出)
- ✅ 符合用户心智模型
- ✅ 便于现场演示 (从左到右)
- ✅ 响应式友好 (移动端可上下堆叠)

### Q5: 色彩方案如何应用？

**建议**: 基于 PRD 色彩规范，采用 **功能导向配色**

**色彩应用指南**:

| 元素 | 颜色 | 色值 | 用途 |
|------|------|------|------|
| 背景 | 浅灰 | #F7F8FA | 页面整体背景 |
| 主色 | 靛蓝 | #4F46E5 | 按钮、链接、强调 |
| 强调色 | 翠绿 | #10B981 | 成功状态、高亮 |
| 文本 | 深灰 | #111827 | 正文、标题 |
| 辅助文本 | 中灰 | #6B7280 | 提示、说明 |
| 卡片背景 | 白 | #FFFFFF | 内容区块 |

**使用原则**:
1. 主色用于 **交互元素** (按钮、链接)
2. 强调色用于 **成功/完成状态**
3. 文本色用于 **内容阅读**
4. 辅助文本用于 **次要信息**
5. 避免超过 3 种主色调

## 🔧 开发实践建议

### Q6: 如何处理 AI 生成的 Loading 状态？

**建议**: 采用 **渐进式 Loading + 文字提示**

**实现方案**:
```
[=====           ] 45%
正在生成项目结构...

↓ 完成后

[==========      ] 75%
正在生成路演稿...

↓ 完成后

[=============   ] 90%
正在生成评委问答...

↓ 完成后

[================] 100%
生成完成！✅
```

**理由**:
- ✅ 用户知道进度 (不会以为卡死)
- ✅ 展示多步骤 (体现工作流价值)
- ✅ 降低等待焦虑
- ✅ 增强科技感

---

### Q7: 如何处理 API 调用失败？

**建议**: 采用 **三层防护机制**

**防护策略**:

1. **前端验证** (第一层)
   - 检查输入是否为空
   - 检查输入长度 (避免超长)
   - 检查格式是否正确

2. **后端处理** (第二层)
   - 捕获 API 异常
   - 返回友好错误信息
   - 记录错误日志

3. **用户体验** (第三层)
   - 显示错误提示 (非技术语言)
   - 提供重试按钮
   - 保留用户输入 (避免重复填写)

**错误提示示例**:
```
❌ 生成失败，请稍后重试。
可能原因：网络不稳定、服务繁忙
[重试按钮] [查看示例输出]
```

### Q8: 如何优化 Prompt 输出质量？

**建议**: 采用 **结构化 Prompt + 示例引导**

**优化技巧**:

1. **明确角色定位**
   ```
   你是一个专业的 AI 项目路演教练，擅长帮助黑客松参与者...
   ```

2. **提供输出模板**
   ```
   请按照以下格式输出：
   ## 一、项目结构
   - 项目名称：xxx
   - 一句话介绍：xxx
   ```

3. **添加约束条件**
   ```
   输出要求：
   - 使用中文
   - 内容务实，不要夸大
   - 不要编造未完成的功能
   ```

4. **提供示例输入输出**
   - 在 Prompt 中包含 1-2 个示例
   - 帮助 AI 理解期望的输出风格

## 🚀 路演演示建议

### Q9: 现场 Demo 如何设计？

**建议**: 采用 **预设输入 + 实时生成** 组合策略

**演示流程** (2-3 分钟):
1. **介绍产品** (30 秒)
   - "我做的是 PitchForge，一个 AI 路演工坊..."
   
2. **展示输入** (30 秒)
   - 打开页面，展示输入区
   - 使用预设示例输入 (避免现场构思)

3. **点击生成** (10 秒)
   - "现在点击生成按钮..."
   - 展示 Loading 状态 (体现工作流)

4. **展示输出** (60 秒)
   - 逐块展示 4 个输出
   - 强调结构清晰、内容务实

5. **总结价值** (20 秒)
   - "这个工具帮助 Builder 在短时间内..."

**备用方案**:
- 准备一份预先生成的输出 (防止网络问题)
- 准备 3 个不同场景的示例 (展示通用性)

### Q10: 如何应对评委追问？

**建议**: 准备 **5 个高频问题** 的标准回答

**高频问题与回答思路**:

1. **Q: 和普通 Chatbot 有什么区别？**
   - A: Chatbot 是自由对话，PitchForge 是结构化流程，输出更稳定、更适合路演直接使用

2. **Q: 为什么一定要用 AI？**
   - A: 人工整理需要 2-3 小时，AI 只需 1-2 分钟，且输出结构标准化

3. **Q: 90 分钟内实际完成了什么？**
   - A: 完整的前后端系统 + 百炼工作流接入 + 可演示的 MVP

4. **Q: 真实用户是谁？**
   - A: 黑客松参与者、AI Workshop 学员、独立开发者、早期项目 Builder

5. **Q: 后续如何扩展？**
   - A: Hackathon Copilot (PRD 生成、README 生成、提交材料、赛后复盘)
